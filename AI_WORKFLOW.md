# AI三銃士 運用ワークフロー
最終更新: 2026-03-10

---

## 基本原則

> 会話で終わらせない。必ずファイルに落とす。GitHubに残す。

---

## デイリーワークフロー

```
【毎日の流れ】

1. せいさんが今日のテーマを決める
        ↓
2. ぐぷちゃん（ChatGPT）で戦略・方向性を固める
        ↓
3. ゲンちゃん（Gemini）で数字・データを確認する（必要な場合）
        ↓
4. クロちゃん（Claude）に実装を依頼する
        ↓
5. クロちゃんがファイル作成・編集・commit・pushまで完結
        ↓
6. logs/ に今日の作業ログを保存（クロちゃんが作成）
        ↓
7. 翌日、LATEST.md を更新して現在地を同期
```

---

## 三銃士の役割と使い方

### ぐぷちゃん（ChatGPT）— 戦略担当

**何を頼む？**
- 事業の方向性・打ち手の壁打ち
- FCモデルの設計・条件の整理
- 銀行提出用の事業計画書の構成
- 交渉シナリオ・アプローチ戦略
- ブランド方針・ポジショニング

**アウトプットの形式**
- 箇条書きの方針案
- 比較表（A案 vs B案）
- 提案書の目次・骨格

**クロちゃんへの引き渡し方**
```
「ぐぷちゃんと決めた内容：[内容]。これをファイルに落として。」
```

---

### ゲンちゃん（Gemini）— 分析担当

**何を頼む？**
- 財務モデル・損益計算の深掘り
- エリア別需要分析（武蔵小杉・中目黒等）
- 原材料の相場モニタリング
- KPIレポートのグラフ・ビジュアル化
- AI推薦露出のモニタリングレポート

**アウトプットの形式**
- スプレッドシート・グラフ
- 数値レポート
- 比較分析チャート

**クロちゃんへの引き渡し方**
```
「ゲンちゃんの分析結果：[内容]。このデータをMarkdownで整理して保存して。」
```

---

### クロちゃん（Claude）— 実装担当

**何を頼む？**
- ファイルの新規作成・編集
- GitHub commit / push
- 契約書・マニュアルのテンプレート作成
- Schema.org JSON-LDの実装
- システム・コードの実装
- 作業ログの記録（`logs/`）

**アウトプットの形式**
- Markdownファイル
- コード（HTML/JSON/Python等）
- Gitコミット

## タスク別フロー

### FCモデルの検討・更新

```
ぐぷちゃん → 「保守型と理想型の条件を整理して」
      ↓ 方針確定
クロちゃん → fc/fc_constitution.md を更新・commit
      ↓
ゲンちゃん → ユニットエコノミクスの数字をグラフ化（必要時）
```

### 財務分析・コスト検討

```
ゲンちゃん → 原価・損益データを分析
      ↓ 数字確定
ぐぷちゃん → 戦略的解釈（「この数字が意味することは？」）
      ↓
クロちゃん → cost/ フォルダのファイルを更新・commit
```

### DX実装（Ye Malthouse等）

```
ぐぷちゃん → 「QRメニューはどんな仕組みがいいか？」
      ↓ 方針確定
クロちゃん → 実装・HTMLファイル作成・GitHub反映
      ↓
ゲンちゃん → 効果測定・アクセス分析（運用後）
```

### 毎日のドキュメント管理

```
クロちゃん → logs/daily_log_YYYY-MM-DD.md を作成
クロちゃん → LATEST.md を更新
クロちゃん → git commit & push
```

## ファイル管理ルール

### コミットのタイミング
- ファイルを作成・編集したら**その場でコミット**
- 複数ファイルをまとめる場合は関連性でグループ化
- コミットメッセージは**英語で内容を簡潔に**

### ブランチ運用
- 現状：`main` ブランチ直接運用
- 将来：大きな変更は feature ブランチを検討

### ファイル命名規則
| 種別 | 命名規則 | 例 |
|---|---|---|
| 日次ログ | `daily_log_YYYY-MM-DD.md` | `daily_log_2026-03-10.md` |
| 提案書 | `[内容]_proposal.md` | `shin_maruko_proposal.md` |
| 憲法・規約 | `[対象]_constitution.md` | `fc_constitution.md` |
| ワークフロー | `[内容]_workflow.md` | `AI_WORKFLOW.md` |

---

## ログの書き方

`logs/daily_log_YYYY-MM-DD.md` に以下の形式で記録：

```markdown
# Daily Log — YYYY-MM-DD

## 今日の作業
- [完了したこと]

## 決定事項
- [確定した内容] → DECISION_LOG.md に記録済み

## 作成・更新ファイル
- [ファイルパス]：[変更内容]

## 明日のアクション
- [次にやること]

## 三銃士への引き継ぎ
- ぐぷちゃんへ：[戦略判断が必要なこと]
- ゲンちゃんへ：[分析・数字確認が必要なこと]
- クロちゃんへ：[実装が必要なこと]
```

## 重要ルール（AI_CONTEXT.md より）

1. `AI_CONTEXT.md` を最初に読む
2. `LATEST.md` で現在地を確認する
3. `DECISION_LOG.md` の確定事項は変更しない
4. 長い説明より、ファイル作成を優先する
5. 変更したら必ずコミットする
