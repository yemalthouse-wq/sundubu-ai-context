# Drop Rules

## Purpose

This file defines the initial manual rules for generating drop from store events.

drop is not money or compensation.
It is a trust signal generated from actions that improved store flow.

---

## Initial Rules

### 1. Peak Support
If a staff member supports operations during `peak_start` and delivery/order flow remains stable, generate drop.

### 2. Recovery Action
If a staff member restores flow after delay, confusion, or operational instability, generate drop.

### 3. Handoff Quality
If a handoff prevents disruption between shifts, generate drop.

### 4. Solo Operation Stability
If solo operation is sustained without collapse during a risky window, generate drop.

### 5. Training Contribution
If training causes short-term cost but improves later operational stability, drop may be generated after validation.

---

## Important Constraints

- drop is additive only
- drop is never removed
- drop is generated from store flow, not personality
- manual judgment comes first, automation later
